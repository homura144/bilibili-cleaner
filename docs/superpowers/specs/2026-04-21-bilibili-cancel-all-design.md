# Bilibili Cleaner 一键取消所有设计说明

## 背景

当前脚本通过给每个条目的“三个点”按钮派发 `mouseover` 事件来展开菜单，再点击“取消追番/追剧”。这种方式依赖页面处于悬浮态，导致用户必须把鼠标停在按钮附近，不能真正做到一键自动执行。

这次改动的目标是把脚本改成从当前页开始自动处理全部追番/追剧条目，并自动翻页直到最后一页，同时移除与该目标无关的旧 UI 和旧流程。

## 目标

- 页面上只保留一个 `一键取消所有` 按钮。
- 点击一次后，脚本自动处理当前页全部条目。
- 当前页完成后，脚本自动进入下一页，直到最后一页。
- 脚本在运行期间显示明确状态，避免重复点击。
- 完成后给出总处理结果提示。

## 非目标

- 不保留作品名打印、复制、下载 TXT、悬浮窗、拖拽等旧功能。
- 不尝试直接调用 B 站内部接口。
- 不处理当前页面以外的入口跳转，只从用户当前打开的追番/追剧列表页开始。

## 方案选择

本次采用“页面自动化”方案，而不是接口逆向方案。

原因如下：

- 页面自动化不再依赖鼠标悬浮，已经能满足“一键取消所有”的需求。
- 直接调用内部接口需要额外处理鉴权、参数和时效性问题，维护成本更高。
- 在当前只有单个 userscript 文件、没有辅助调试基建的前提下，页面自动化更容易落地和回归验证。

## 交互设计

- 在现有过滤栏附近插入一个按钮，文案为 `一键取消所有`。
- 点击后按钮进入运行态，显示如 `取消中...`，并禁用再次点击。
- 运行结束后按钮恢复可点击状态。
- 完成时使用 `alert` 汇总结果，例如总处理条目数、已翻页数、失败数。

## 执行流程

1. 用户进入 B 站追番/追剧页面。
2. 脚本注入 `一键取消所有` 按钮。
3. 用户点击按钮后，脚本锁定运行状态。
4. 脚本扫描当前页条目列表，逐个处理：
   - 定位条目的更多操作按钮。
   - 主动展开菜单，不依赖真实鼠标悬浮。
   - 在菜单中寻找“取消追番”或“取消追剧”动作。
   - 如页面出现确认弹窗，则自动点击确认。
   - 等待当前条目从列表中移除，或等待 DOM 稳定后继续下一项。
5. 当前页处理完成后，判断是否存在可点击的“下一页”按钮。
6. 如果存在，则进入下一页并等待列表刷新后继续处理。
7. 如果不存在，则结束流程并提示结果。

## DOM 与鲁棒性策略

- 选择器优先复用现有脚本已使用的稳定节点，例如标题区域、过滤栏、更多按钮图标。
- 取消动作的匹配优先使用按钮文案关键词 `取消追`，同时兼容“取消追番”和“取消追剧”。
- 翻页逻辑优先按分页控件文案或禁用状态判断，而不是依赖固定页码结构。
- 每个关键步骤都加入短暂轮询等待，避免因为菜单、弹窗或翻页动画尚未完成导致误判。
- 如果某个条目失败，记录失败次数并继续后续条目，避免整批处理中断。

## 错误处理

- 如果当前页未找到任何可处理条目，直接结束并提示。
- 如果菜单未成功展开或未找到取消按钮，记录该条目失败并继续。
- 如果翻页后页面未在超时时间内刷新，终止流程并提示用户检查页面状态。
- 如果用户在执行中离开页面或 DOM 结构变化明显，允许脚本在当前步骤失败后安全停止。

## 测试与验证

- 手工验证按钮只注入一次，不会重复创建。
- 手工验证点击一次后无需鼠标悬浮即可开始处理。
- 手工验证当前页多个条目能连续取消。
- 手工验证处理完当前页后能自动进入下一页。
- 手工验证到最后一页后能自动停止并给出提示。
- 手工验证异常场景下不会因为单个条目失败而整批卡死。

## 实施边界

- 只修改现有的 `BiliBili Cleaner.js`。
- 以最小改动删除旧 UI 和旧逻辑，再补充新的批处理与翻页逻辑。
